You are building a complete **frontend-only static web app MVP** for a school training application.

# PROJECT TITLE

Office Skills Training App

# PROJECT PURPOSE

Build a static training web app that helps **Don Bosco Erdem TVET School** students of the **Office Administration program (1st to 3rd year)** study, review, and practice lessons.

# IMPORTANT CONSTRAINTS

This is a strict MVP with hard limits:

1. **No backend**
2. **No database**
3. **No server-side authentication**
4. **No dynamic site**
5. **No paid tools or services**
6. **Everything must work as a static frontend web app**
7. User session and progress must be stored only on the **client side** using **localStorage**
8. The app must be simple enough to finish within **7 days**
9. The app must look polished enough for a school presentation/demo
10. Use **mock data / local JSON / local JS objects** for lessons and exercises

# TECH DIRECTION

Build this as a clean static web app using one of these options:

* Preferred: **React + Vite**
* Allowed: plain HTML/CSS/JavaScript if simpler
  Choose the option that gives the most polished result quickly.

Use:

* Responsive layout
* Clean modern UI
* Fast loading
* Simple state management
* Easy-to-edit content structure

# GLOBAL APP RULES

## 1. Authentication rule

Because there is no backend, do NOT implement real Google, Apple, or server auth.

Instead:

* Show auth provider buttons for UI polish
* But actual working login must be:

  * “Continue with email” opens a simple local form
  * User enters name
  * Optional email field can be included
  * Save user info to localStorage
  * Redirect to Dashboard

Also:

* Add Logout button
* Logout clears local session from localStorage

## 2. Progress rule

Store these in localStorage:

* current user name
* login state
* lesson progress
* completed exercises
* score history
* current status labels

## 3. Content rule

Use editable local mock content.
Create sample lessons and sample exercises so the app is demo-ready without backend.

## 4. UI rule

The UI must feel modern, academic, and clean.
Do NOT make it childish or overly playful.
Use a light theme.
Good whitespace.
Rounded cards.
Subtle shadows.
Clear typography.
Responsive layout for desktop and mobile.

## PAGE 5 — EXERCISE PAGE

### Goal

Run the actual interactive exercise.

### Global requirements

* Show clear instructions
* Show question/content area
* Show answer interaction area
* Show submit/check action
* Show retry/reset action
* Show progress indicator
* After submitting, go to Result page

### Required exercise types

Implement at least **2 exercise types fully working**.
If possible, scaffold all 4.
Priority order:

### Exercise Type A — Definition Matching / Sentence Ordering

This is the highest priority.

Behavior:

* A long definition is split into 2–3 sentence chunks
* Chunks appear in a **selection bank**
* User clicks chunks to place them into the **answer board**
* If user clicks a placed chunk again, it returns to the bank
* User can reorder by removing and placing again, or via move buttons if easier
* On submit, compare with the correct order
* Then show result

### Exercise Type B — “Би хэн бэ?” / Identity Guess

Behavior:

* Show a clue/description
* Show multiple choices
* User picks the correct answer
* Submit
* Result page shows correct/incorrect with explanation

### Exercise Type C — Legal Case

Optional if time allows

* Show short scenario
* Multiple choice or category selection

### Exercise Type D — Document Components

Optional if time allows

* Match a document with its correct components

### Important UX behavior

* The user must always understand:

  * what the task is
  * what they selected
  * whether they can change it
* Add instruction text near the top
